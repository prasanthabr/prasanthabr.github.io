Converting Salesforce ID From 15digit to 18digit

Jul 17th, 2014 12:52 am

understanding & leveraging record ids

A salesforce.com id field is an auto generated 15 character base-62 id consisting of 26 lower case characters (a-z), 26 upper case characters (A-Z) and 10 numerals (0-9) that remains unique within the organisation data and constant throughout the lifetime of the record, analogous to a key value in a relational database. The first three characters of the id refer to the object to which the record belongs. The remaining 12 characters are said to identify the record id, while the first character of this 12 reference the salesforce.com pod on which the record is created.

It is now well documented that while this system works well inside the salesforce.com ecosystem, salesforce.com admins face issues on oft used third party tools due to lack of support for case sensitivity. Salesforce.com has responded with a case insensitive 18 character id to resolve the situation and have recently made available a CASESAFEID (Id) function for easy conversion of case sensitive ids. While API calls return an 18 character id and salesforce.com recommends using the 18 character id, the salesforce.com reports continue to return a 15 digit id. Attempts have also been made to create workarounds to make third party tools case sensitive.

While these fixes and solutions work in many situations, for the average salesforce.com admin the process remains fraught with complexity. Usage of reports is more common than the api supported dataloader, due to a premium on code footprint and field count, the casesafeid function is seldom used, leaving admins to get creative when working with data.

A common solution is to use the case sensitive vlookup on a csv file. The solution hinges on a case sensitive comparison using the EXACT function in MS Excel, which is prone to deliver a no match found verdict if the raw data is unsorted and hence may be unreliable. Of course, lot of developers have developed solutions in JavaScript, VBA and Apex to solve the issue. The excel connector bestows a custom function that can easily convert a 15 digit id to an 18 digit one.

The idea itself behind converting the case sensitive id to a case insensitive one is quite simple. The 15 digit id is appended with three extra characters which would identify the position of cases in the strings. Though commonly mistaken as a checksum, the three characters are not checksums since these three characters are not unique to the id and hence would not be able to identify an id corruption.

Though there does not seem to exist an official documentation on the conversion, the algorithm is commonly available. The solution calls for slicing the 15 digit id to three parts. Based on the location of upper cases in these slices, a 5 digit binary code is generated. The generated binary code picks up a matching character from the 62 count character list, appending the three generated characters to the original 15 character id.
